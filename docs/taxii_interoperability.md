## 1. Executive Summary & System Context

The **PhantomNet TAXII 2.1 Feed Server** provides an OASIS TAXII 2.1 standard-compliant REST API for distributing machine-readable STIX 2.1 threat intelligence generated by the PhantomNet Sentinel Layer.

## 2. Protocol Interoperability & Media-Type Negotiation Contract

TAXII 2.1 (OASIS Standard) mandates explicit MIME media-type negotiation via HTTP `Accept` and `Content-Type` headers. PhantomNet strictly enforces these rules at both middleware and route handler levels.

### 2.1 Standard Media Types

| Asset Type | OASIS Standard Media Type Header |
| :--- | :--- |
| **TAXII 2.1 Metadata** | `application/taxii+json;version=2.1` |
| **STIX 2.1 Objects** | `application/stix+json;version=2.1` |

### 2.2 Header Negotiation & Validation Rules

1. **`Accept` Header Requirements:**
   - **Discovery, API Root, and Collections endpoints** require `Accept: application/taxii+json;version=2.1` (or wildcards `application/*`, `*/*`).
   - **Objects endpoints** accept `Accept: application/stix+json;version=2.1` or `application/taxii+json;version=2.1`.
   - If an explicit `Accept` header specifies an unsupported format (e.g. `text/html`, `application/xml`) or an invalid version parameter (e.g. `version=2.0`), the server returns HTTP **`406 Not Acceptable`**.

2. **`Content-Type` Header Requirements:**
   - Client requests supplying `Content-Type` headers MUST specify `application/taxii+json`, `application/stix+json`, or `application/json` with optional `;version=2.1`. Requests specifying unsupported versions return HTTP **`406 Not Acceptable`**.

3. **Response `Content-Type` Headers:**
   - Metadata endpoint responses return `Content-Type: application/taxii+json;version=2.1`.
   - Object endpoint responses return `Content-Type: application/stix+json;version=2.1` (or `application/taxii+json;version=2.1` if requested).

### 2.3 HTTP Status Codes & Error Contract

All error responses return a standardized `TaxiiErrorResponse` body formatted per TAXII 2.1 §3.2 with `Content-Type: application/taxii+json;version=2.1`:

```json
{
  "title": "Not Acceptable",
  "description": "The requested Accept header 'application/xml' is not supported by TAXII 2.1. Expected 'application/taxii+json;version=2.1'.",
  "http_status": "406"
}
```

| HTTP Status Code | Condition | Description |
| :--- | :--- | :--- |
| **`200 OK`** | Success | Request succeeded and returned valid TAXII metadata or STIX bundle. |
| **`400 Bad Request`** | Invalid Query | `added_after` parameter is malformed or invalid ISO 8601 string. |
| **`404 Not Found`** | Resource Missing | Requested collection ID or endpoint does not exist. |
| **`406 Not Acceptable`** | Media Negotiation Error | Unsupported `Accept` or `Content-Type` header supplied. |
| **`405 Method Not Allowed`** | Invalid HTTP Method | Non-GET HTTP methods (e.g., `POST`, `PUT`, `DELETE`) on read-only feed. |

## 3. Endpoints Specification & Response Payloads

All TAXII endpoints are served under the `/taxii2` prefix. Both trailing slash and non-trailing slash URLs are fully supported.

- **Purpose:** Discovers available API Roots and server metadata.
- **Header:** `Accept: application/taxii+json;version=2.1`
- **Response `200 OK` Payload:**

## 4. Feed Integration Architecture & STIX Mapping

The feed integration pipeline automatically maps internal Sentinel Playbook database entities to standard STIX 2.1 objects:

1. **`SentinelPlaybook` Record:** Contains `playbook_id`, `playbook_name`, `tactic`, `technique`, `src_ip`, `dst_port`, and timestamp fields.
2. **STIX 2.1 Report Object:** Generated for each playbook, serving as the container metadata holding description, published timestamp, and references to associated indicators.
3. **STIX 2.1 Indicator Object:** Generated when a malicious `src_ip` is present. Uses standard STIX pattern syntax `[ipv4-addr:value = 'x.x.x.x']`.
4. **Dynamic Collection Filtering:**
   - `sentinel-playbooks-approved` -> Returns all approved playbooks.
   - `tactic-{slug}` -> Filters playbooks by MITRE ATT&CK tactic (e.g. `tactic-initial-access`).
   - `honeypot-{service}` -> Filters playbooks by honeypot destination port (e.g. port 22 -> `honeypot-cowrie-ssh`).

## 5. Client Interoperability & Integration Guide

### 5.1 Python `taxii2-client` Integration Example

PhantomNet provides a built-in TAXII 2.1 client wrapper (`backend/api/taxii_client.py`). External Python applications can also connect using the official OASIS `taxii2-client` package:

```python
from taxii2client.v21 import Server, Collection

# 1. Connect to PhantomNet TAXII 2.1 Discovery Endpoint
server = Server("http://localhost:8000/taxii2/")
print(f"Connected to: {server.title}")

# 2. Get Default API Root & List Collections
api_root = server.api_roots[0]
collections = api_root.collections

for col in collections:
    print(f"Collection ID: {col.id} | Title: {col.title}")

# 3. Retrieve Objects with Incremental Filtering
target_collection = api_root.get_collection("sentinel-playbooks-approved")
stix_bundle = target_collection.get_objects(added_after="2026-07-01T00:00:00Z")

print(f"Retrieved {len(stix_bundle.get('objects', []))} STIX 2.1 objects")
```

### 5.2 cURL CLI Integration Examples

```bash
# 1. Discovery Endpoint
curl -X GET "http://localhost:8000/taxii2/" \
  -H "Accept: application/taxii+json;version=2.1"

# 2. Collections List
curl -X GET "http://localhost:8000/taxii2/phantomnet/collections/" \
  -H "Accept: application/taxii+json;version=2.1"

# 3. Retrieve STIX Objects filtered by added_after timestamp
curl -X GET "http://localhost:8000/taxii2/phantomnet/collections/sentinel-playbooks-approved/objects/?added_after=2026-07-24T00:00:00Z" \
  -H "Accept: application/stix+json;version=2.1"
```
